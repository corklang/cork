# MOS 6510 CPU Reference

A comprehensive technical reference for the MOS Technology 6510 microprocessor
as used in the Commodore 64 home computer.

---

## 1. Overview

The **MOS Technology 6510** is an 8-bit microprocessor and a direct descendant of
the legendary MOS 6502. Designed by MOS Technology (later acquired by Commodore
Semiconductor Group), the 6510 is the CPU at the heart of the Commodore 64, the
best-selling home computer of all time.

### Key Specifications

| Parameter          | Value                              |
|--------------------|------------------------------------|
| Architecture       | 8-bit                              |
| Data bus           | 8 bits                             |
| Address bus        | 16 bits (64 KB addressable)        |
| Clock speed (NTSC) | 1.022727 MHz                      |
| Clock speed (PAL)  | 0.985248 MHz                      |
| Package            | 40-pin DIP                         |
| Transistor count   | ~3,510 (NMOS process)             |
| Instruction count  | 56 official mnemonics, 151 opcode variants |
| Addressing modes   | 13                                 |
| Stack              | 256 bytes, hardwired at $0100-$01FF |
| Endianness         | Little-endian                      |

### What Makes the 6510 Special

The 6510 is instruction-set compatible with the 6502 -- every 6502 program runs
unchanged on a 6510, including undocumented opcodes. The key difference is the
addition of a **built-in 6-bit bidirectional I/O port** mapped to addresses
**$00** and **$01** in zero page. This port is the mechanism by which the
Commodore 64 implements bank switching, allowing its 64 KB of RAM to coexist
with ROMs and I/O hardware in the same address space.

The 6510 also adds tri-state control over the address bus and a clean halt
capability, neither of which the original 6502 possessed. The designers
repurposed three previously unconnected pins (5, 35, 36) and rearranged others
to free six contiguous pins (24-29) for the parallel I/O port lines P0-P5.

### The Processor Port ($00-$01)

**$0000 -- Data Direction Register (DDR)**

Each bit controls whether the corresponding port pin is an input (0) or an
output (1). On the C64, the default DDR value is `$2F` (`%00101111`), making
bits 0-3 and 5 outputs, and bits 4 and 6-7 inputs (bits 6-7 are unconnected).

**$0001 -- Processor Port (Data Register)**

The actual I/O port. On the C64, the default value is `$37` (`%00110111`).

| Bit | Name    | Direction | Function                                      |
|-----|---------|-----------|-----------------------------------------------|
| 0   | LORAM   | Output    | 1 = BASIC ROM visible at $A000-$BFFF          |
| 1   | HIRAM   | Output    | 1 = KERNAL ROM visible at $E000-$FFFF         |
| 2   | CHAREN  | Output    | 1 = I/O at $D000-$DFFF; 0 = Character ROM     |
| 3   | CASS WR | Output    | Datasette write data line                      |
| 4   | CASS SENSE | Input  | Datasette button sense (active low)            |
| 5   | CASS MOTOR | Output | Datasette motor control (1 = motor off)        |
| 6-7 | --      | --        | Not connected on standard C64                  |

### Bank Switching Memory Map

Bits 0-2 of the processor port feed into the C64's PLA (Programmable Logic
Array), which controls what the CPU sees in three contested memory regions:

| LORAM | HIRAM | CHAREN | $A000-$BFFF | $D000-$DFFF | $E000-$FFFF |
|:-----:|:-----:|:------:|:-----------:|:-----------:|:-----------:|
|   1   |   1   |   1    | BASIC ROM   | I/O         | KERNAL ROM  |
|   0   |   1   |   1    | RAM         | I/O         | KERNAL ROM  |
|   1   |   0   |   1    | RAM         | I/O         | RAM         |
|   0   |   0   |   1    | RAM         | I/O         | RAM         |
|   1   |   1   |   0    | BASIC ROM   | Char ROM    | KERNAL ROM  |
|   0   |   1   |   0    | RAM         | Char ROM    | KERNAL ROM  |
|   1   |   0   |   0    | RAM         | Char ROM    | RAM         |
|   0   |   0   |   0    | RAM         | RAM         | RAM         |

Note: When HIRAM=0, BASIC ROM at $A000-$BFFF is always replaced by RAM
regardless of the LORAM bit. The region $0002-$9FFF and $C000-$CFFF is always
RAM. When bits 0-1 are both 0, the I/O region at $D000-$DFFF becomes RAM
regardless of CHAREN.

## 2. Registers

The 6510 has the same register set as the 6502: five 8-bit registers and one
16-bit register.

### Accumulator (A) -- 8 bits

The primary register for arithmetic and logic operations. Most ALU instructions
operate on the accumulator: ADC, SBC, AND, ORA, EOR, ASL, LSR, ROL, ROR. Data
flows through A for nearly all computation.

### Index Register X (X) -- 8 bits

Used for indexed addressing modes (Zero Page,X and Absolute,X), as the offset
in Indexed Indirect (Indirect,X) addressing, and for loop counting. Also used
to transfer values to/from the stack pointer (TXS/TSX).

### Index Register Y (Y) -- 8 bits

Used for indexed addressing modes (Zero Page,Y and Absolute,Y), and as the
offset in Indirect Indexed (Indirect),Y addressing. Somewhat less versatile
than X since it cannot directly set the stack pointer.

### Stack Pointer (SP) -- 8 bits

Points to the next free location on the stack. The stack is hardwired to page 1
($0100-$01FF) and grows downward: a push decrements SP, a pull increments it.
On reset, SP is initialized to $FD (after the reset sequence pushes three
dummy bytes). The effective stack address is always $0100 + SP.

### Program Counter (PC) -- 16 bits

The only 16-bit register. Holds the address of the next instruction to be
fetched. Automatically incremented during instruction execution. Modified by
branch, jump, subroutine, interrupt, and return instructions.

### Processor Status Register (P) -- 8 bits

```
  Bit 7   Bit 6   Bit 5   Bit 4   Bit 3   Bit 2   Bit 1   Bit 0
+-------+-------+-------+-------+-------+-------+-------+-------+
|   N   |   V   |   -   |   B   |   D   |   I   |   Z   |   C   |
+-------+-------+-------+-------+-------+-------+-------+-------+
```

| Bit | Flag | Name      | Description                                          |
|-----|------|-----------|------------------------------------------------------|
| 7   | N    | Negative  | Set if bit 7 of the result is set (sign bit). Reflects whether the result is negative in two's complement. |
| 6   | V    | Overflow  | Set if signed arithmetic overflow occurred. For ADC/SBC, set when the sign of the result is wrong given the signs of the operands. For BIT, set to bit 6 of the memory operand. |
| 5   | -    | (Unused)  | Always reads as 1. Not a real flag; exists only as a bit position in the status register. When pushed to the stack (PHP or interrupt), this bit is always set. |
| 4   | B    | Break     | Not a physical flag in the register. Distinguishes software interrupts (BRK, B=1) from hardware interrupts (IRQ/NMI, B=0) when the status register is pushed to the stack. Only exists in the pushed copy, never in the actual register. |
| 3   | D    | Decimal   | When set, ADC and SBC operate in BCD (Binary-Coded Decimal) mode. On NMOS 6502/6510, the D flag is NOT cleared on reset or interrupt -- the programmer must explicitly clear it with CLD. |
| 2   | I    | Interrupt Disable | When set, maskable interrupts (IRQ) are disabled. Set automatically during interrupt processing and by SEI. Cleared by CLI and RTI. Does NOT affect NMI. |
| 1   | Z    | Zero      | Set if the result of the last operation was zero. |
| 0   | C    | Carry     | Carry out from bit 7 for addition (ADC), borrow (inverted) for subtraction (SBC), and the bit shifted out for shift/rotate instructions. Used as input for ADC, SBC, ROL, and ROR. |

## 3. Addressing Modes

The 6510 supports 13 addressing modes. Not every instruction supports every
mode; each instruction has its own subset.

### Implied (Implicit)

The instruction operates on a specific register or performs a fixed operation;
no operand is needed.

```
  CLC          ; Clear carry flag
  INX          ; Increment X register
  RTS          ; Return from subroutine
```

**Size:** 1 byte (opcode only)

### Accumulator

The instruction operates on the accumulator. In some assemblers written as
`ASL A` or just `ASL` with no operand.

```
  ASL A        ; Shift accumulator left
  ROR A        ; Rotate accumulator right
```

**Size:** 1 byte (opcode only)

### Immediate

The operand is a literal 8-bit value embedded in the instruction stream,
denoted by `#`.

```
  LDA #$42     ; Load $42 into A
  AND #$0F     ; AND accumulator with $0F
  CPX #$00     ; Compare X with zero
```

**Size:** 2 bytes (opcode + 1-byte value)

### Zero Page

The operand is an 8-bit address in zero page ($0000-$00FF). One byte shorter
and one cycle faster than absolute addressing, making zero page the prime
location for frequently accessed variables.

```
  LDA $FB      ; Load from address $00FB
  STA $02      ; Store to address $0002
```

**Size:** 2 bytes (opcode + 1-byte address)

### Zero Page, X

Zero-page address plus the X register, with wraparound within zero page. If
the address + X exceeds $FF, it wraps to $00 (no page crossing).

```
  LDA $80,X    ; Load from ($80 + X) AND $FF
  STA $10,X    ; Store to ($10 + X) AND $FF
```

**Size:** 2 bytes (opcode + 1-byte base address)

### Zero Page, Y

Same as Zero Page,X but using the Y register. Only available for LDX and STX
(official opcodes).

```
  LDX $80,Y    ; Load X from ($80 + Y) AND $FF
  STX $10,Y    ; Store X to ($10 + Y) AND $FF
```

**Size:** 2 bytes (opcode + 1-byte base address)

### Absolute

The full 16-bit address is specified in the instruction. Stored little-endian
(low byte first).

```
  LDA $D020    ; Load from VIC border color register
  JMP $FCE2    ; Jump to KERNAL reset routine
  STA $0400    ; Store to screen memory
```

**Size:** 3 bytes (opcode + 2-byte address)

### Absolute, X

A 16-bit base address plus the X register. If adding X causes the low byte to
cross a page boundary (carry from bit 7 to bit 8), an extra cycle is consumed
for the page-fix read. Write instructions always take the extra cycle.

```
  LDA $0400,X  ; Load from screen memory + X
  STA $D800,X  ; Store to color RAM + X
```

**Size:** 3 bytes (opcode + 2-byte base address)

### Absolute, Y

Same as Absolute,X but using the Y register. Same page-crossing penalty rules
apply.

```
  LDA $C000,Y  ; Load from $C000 + Y
  STA $0400,Y  ; Store to screen memory + Y
```

**Size:** 3 bytes (opcode + 2-byte base address)

### Indirect

Used exclusively by JMP. A 16-bit address points to a memory location
containing the actual 16-bit target address (the "vector").

```
  JMP ($0314)  ; Jump through the IRQ vector
```

**Size:** 3 bytes (opcode + 2-byte pointer address)

**BUG:** If the pointer address ends in $FF (e.g., `JMP ($02FF)`), the CPU
fetches the low byte from $02FF but wraps within the page and fetches the high
byte from $0200 instead of $0300. See Section 6.

### Indexed Indirect -- (Indirect, X)

An 8-bit zero-page address is added to X (with zero-page wraparound) to form
a pointer. The CPU reads the 16-bit target address from that pointer location.
Used for table-driven dispatch or accessing arrays of pointers.

```
  LDA ($20,X)  ; Read pointer from ($20+X) & $FF,
               ; then load from address in pointer
```

**Size:** 2 bytes (opcode + 1-byte zero-page base)

Cycle breakdown: opcode fetch, base byte fetch, add X (discarded read), read
pointer low, read pointer high, read from effective address = 6 cycles.

### Indirect Indexed -- (Indirect), Y

An 8-bit zero-page address points to a 16-bit base address. The Y register is
added to that base address to form the effective address. The most commonly used
indirect mode -- ideal for traversing data structures.

```
  LDA ($FB),Y  ; Read pointer from $FB/$FC,
               ; add Y, load from result
  STA ($22),Y  ; Read pointer from $22/$23,
               ; add Y, store to result
```

**Size:** 2 bytes (opcode + 1-byte zero-page pointer)

Page-crossing penalty: +1 cycle if (pointer_low + Y) > $FF for read
instructions. Write instructions always take the extra cycle.

### Relative

Used exclusively by branch instructions. The operand is a signed 8-bit offset
(-128 to +127) added to the PC (specifically, to the address of the byte
following the branch instruction).

```
  BNE label    ; Branch if Z=0: PC = PC + 2 + offset
  BEQ $FA      ; Branch backward 6 bytes if Z=1
```

**Size:** 2 bytes (opcode + 1-byte signed offset)

Cycle behavior:
- Branch not taken: 2 cycles
- Branch taken, no page cross: 3 cycles
- Branch taken, page cross: 4 cycles

- `*` = add 1 cycle if page boundary crossed

## 5. Undocumented / Illegal Opcodes

The NMOS 6502/6510 has 256 possible opcodes but only 151 are officially
documented. The remaining 105 are "illegal" or "undocumented" opcodes. Many of
these perform useful operations and are actively used in C64 demos, cracktros,
and games. They arise from how the 6502's PLA decodes overlapping instruction
patterns.

**Stability classification:**
- **Stable** -- Behaves consistently and predictably on all NMOS 6502/6510 chips.
- **Unstable** -- Behavior depends on analog effects, data bus contents, or
  timing; may vary between chips or even between runs.
- **Highly unstable** -- Involves bus contention; results are essentially
  unpredictable.

## 6. Hardcore Details

### 6.1 Cycle-Exact Timing and Pipeline Behavior

The 6510 does not have a pipeline in the modern sense, but its internal
sequencing overlaps instruction boundaries. The last cycle of every instruction
is simultaneously the opcode fetch for the next instruction. This overlap is
what creates the "minimum 2 cycles per instruction" constraint.

**T-States:** Internally, the 6510 uses timing states T0 through T5 (plus
special states for RMW instructions) to sequence each instruction:

- **T0:** Last cycle of current instruction / first cycle of next (opcode fetch).
  The SYNC pin is asserted during T1.
- **T1:** Opcode held in Instruction Register; operand byte fetched.
- **T2-T5:** Progressive execution phases depending on the instruction.
- **SD1, SD2:** Independent timing chain for Read-Modify-Write operations.

**Two-cycle instructions** (e.g., NOP, CLC, TAX): Compressed to just T0 and
the overlapped next-instruction fetch. The sequence is `T0 -> T2 -> (next T0)`.

**Standard cycle patterns by addressing mode:**

| Addressing Mode     | Read  | Write | RMW   |
|---------------------|-------|-------|-------|
| Implied             | 2     | --    | --    |
| Accumulator         | 2     | --    | --    |
| Immediate           | 2     | --    | --    |
| Zero Page           | 3     | 3     | 5     |
| Zero Page, X/Y      | 4     | 4     | 6     |
| Absolute            | 4     | 4     | 6     |
| Absolute, X/Y       | 4+p   | 5     | 7     |
| (Indirect, X)       | 6     | 6     | 8     |
| (Indirect), Y       | 5+p   | 6     | 8     |
| Relative (not taken) | 2    | --    | --    |
| Relative (taken)     | 3+p  | --    | --    |

Where `p` = 1 if a page boundary is crossed, 0 otherwise.

### 6.2 Page-Crossing Penalties

A page crossing occurs when adding an index register to a base address causes
the high byte to change (i.e., carry from bit 7 to bit 8 of the address low
byte).

**Read instructions:** The CPU speculatively reads from the uncorrected address
(wrong page) and must spend an additional cycle to fix the high byte and re-read
from the correct address. This adds 1 cycle.

**Write instructions (STA, etc.):** Always take the extra cycle regardless of
whether a page boundary was actually crossed. The CPU always performs the
dummy read before writing.

**RMW instructions (illegal, e.g., DCP abs,X):** Always take the maximum cycle
count. No conditional penalty; the extra cycle is always present.

**Branch instructions:** A taken branch adds 1 cycle. If the branch target is
on a different page than the instruction following the branch, an additional
cycle is added for the page-crossing fix (total: +2 cycles over not-taken).

### 6.3 Read-Modify-Write Behavior

RMW instructions (INC, DEC, ASL, LSR, ROL, ROR and their illegal combined
variants) perform a sequence of: read the value, write the original value back,
then write the modified value. This "double write" means:

1. The original (unmodified) value is written to the address first.
2. Then the modified value is written.

This has important consequences for hardware registers: writing to a VIC-II or
SID register with an RMW instruction will briefly glitch the original value
before setting the new one. INC $D019 to acknowledge a VIC raster IRQ works
specifically because of this: the first write clears the flag, and the second
write sets it again, but the acknowledgment has already happened.

### 6.4 The JMP ($xxFF) Indirect Bug

The JMP instruction's indirect addressing mode has a well-known hardware bug.
When the pointer address falls on a page boundary (low byte = $FF), the CPU
wraps within the page for the high byte of the target address instead of
crossing to the next page.

Example:
```
  ; Memory contents:
  ;   $02FF = $80
  ;   $0300 = $50
  ;   $0200 = $40

  JMP ($02FF)
  ; Expected: fetch low from $02FF ($80), high from $0300 ($50) -> jump to $5080
  ; Actual:   fetch low from $02FF ($80), high from $0200 ($40) -> jump to $4080
```

The CPU correctly reads the low byte from $02FF, but wraps to $0200 for the
high byte instead of advancing to $0300. This is because the 6502 increments
only the low byte of the pointer address internally, and the carry is never
propagated to the high byte. This bug is fixed in the CMOS 65C02.

**Workaround:** Never place an indirect jump vector at a $xxFF address. Ensure
vectors are aligned to not straddle page boundaries.

### 6.5 The BRK / IRQ Conflict

If a hardware interrupt (IRQ or NMI) occurs during the execution of a BRK
instruction, the BRK is effectively "hijacked." The pushed return address is
still that of the BRK (PC+2), but the interrupt vector used is that of the
hardware interrupt (NMI vector $FFFA or IRQ vector $FFFE), not the BRK vector.
Since BRK and IRQ share the same vector ($FFFE), this is only observably
different when NMI hijacks BRK.

Additionally, the B flag in the pushed status register may be 0 (indicating
hardware interrupt) even though a BRK instruction was executing. Software
relying on the B flag to distinguish BRK from IRQ must account for this race
condition.

### 6.6 Interrupt Handling

#### Types of Interrupts

| Interrupt | Vector     | Trigger       | Maskable | Priority |
|-----------|------------|---------------|----------|----------|
| NMI       | $FFFA-$FFFB | Falling edge | No       | Highest  |
| RESET     | $FFFC-$FFFD | Low level    | No       | Special  |
| IRQ/BRK   | $FFFE-$FFFF | Low level / opcode | Yes (IRQ) | Lowest |

#### Interrupt Sequence (7 cycles for NMI/IRQ)

```
  Cycle 1: Read opcode of next instruction (discarded)
  Cycle 2: Read operand byte (discarded)
  Cycle 3: Push PCH to stack
  Cycle 4: Push PCL to stack
  Cycle 5: Push P to stack (B=0 for hardware, B=1 for BRK)
  Cycle 6: Read vector low byte
  Cycle 7: Read vector high byte -> PCH
           I flag is set
```

#### IRQ (Interrupt Request)

- Level-sensitive: the IRQ line must be held low for the interrupt to be
  detected.
- Masked when the I flag is set (SEI). Cleared by CLI or RTI.
- Polled during the last cycle of each instruction (specifically, during
  phase 2 of the penultimate cycle). If low, the interrupt sequence begins
  instead of the next instruction.
- IRQ latency: the current instruction always completes first. Minimum
  latency from assertion to first cycle of handler = 2 cycles (if detected
  during a 2-cycle instruction) + 7 cycles for the interrupt sequence = 9
  cycles. Maximum latency depends on the longest instruction running (7
  cycles for an illegal RMW instruction) + 7 = 14 cycles.

#### NMI (Non-Maskable Interrupt)

- Edge-sensitive: triggered by a high-to-low transition on the NMI pin.
- Cannot be masked by the I flag.
- The edge detector samples during phase 2 of each cycle. If the NMI line
  transitions from high to low, an internal NMI-pending flag is raised during
  phase 1 of the following cycle.
- NMI takes priority over IRQ if both are pending simultaneously.
- A known quirk: if the NMI line goes low and then high again before the NMI
  is serviced, a second falling edge can trigger another NMI. Holding NMI low
  continuously only triggers one NMI.

#### Branch Instruction and Interrupt Polling

Branch instructions have special interrupt polling behavior. For most
instructions, interrupt polling occurs at the T0 state (last cycle). But:

- **Branch not taken:** Polling occurs at T2 (the end of the 2-cycle
  instruction), equivalent to T0 for non-branch instructions.
- **Branch taken, no page cross:** Polling does NOT occur at the end of the
  branch. The interrupt is delayed until the end of the following instruction.
- **Branch taken, page cross:** Polling occurs during the page-fix cycle.

This means that a tight branch loop like `here: JMP here` or
`here: BNE here` can delay interrupt response by one instruction.

### 6.7 Decimal Mode (BCD) on NMOS 6510

When the D flag is set (SED), ADC and SBC perform Binary-Coded Decimal
arithmetic. Each nibble (4 bits) represents a decimal digit 0-9.

**Key NMOS 6510 characteristics:**

- Only ADC and SBC are affected by decimal mode. All other instructions ignore
  the D flag entirely.
- **ADC in decimal mode:** Both operands are treated as BCD. The result in A
  and the C flag are correct BCD values. However, the N, V, and Z flags
  reflect the intermediate binary result BEFORE decimal correction is applied.
  This means Z cannot reliably detect a zero result in BCD mode on NMOS.
- **SBC in decimal mode:** The result in A and C are correct. As with ADC,
  N, V, Z reflect the binary subtraction, not the BCD result.
- **Invalid BCD digits:** If the operands contain nibbles > 9 (e.g., $0A-$0F),
  the processor still performs the BCD fixup, producing unpredictable (but
  deterministic) results.
- **The D flag is NOT cleared on reset, IRQ, or NMI** on NMOS chips. Good
  practice requires a CLD instruction at the start of every interrupt handler
  and after reset. (The C64 KERNAL does this.)
- **CMOS 65C02 differences:** The 65C02 fixes the flag behavior -- N, V, Z
  are all valid after BCD operations. However, BCD ADC/SBC take one extra
  cycle (3 cycles for immediate instead of 2). The 65C02 also clears D
  automatically on interrupt and reset.

### 6.8 Reset Behavior

On reset (RESET pin held low), the 6510 performs the following 7-cycle
sequence:

```
  Cycle 1-3: Internal operations (similar to interrupt but writes are
             suppressed -- the stack pushes become reads instead)
  Cycle 4-5: Stack pointer decremented 3 times (SP -= 3) without writing
  Cycle 6: Read vector low byte from $FFFC
  Cycle 7: Read vector high byte from $FFFD -> PC
           I flag is set
```

The key detail: the stack pushes during reset are converted to reads, so the
stack contents are not modified. The SP is still decremented by 3 (from its
previous value). After reset, if SP was previously $FF, it will be $FC.
Typical post-reset SP value is $FD.
